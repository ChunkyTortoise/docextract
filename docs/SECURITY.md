# Security Guide

DocExtract uses DB-backed API keys, role-aware authorization, Redis rate limiting, signed webhooks, and PII sanitization for external traces.

## API Authentication

Clients authenticate with the `X-API-Key` header. API keys are generated by `POST /api/v1/api-keys`, returned in plaintext once, HMAC-SHA256 hashed with `API_KEY_SECRET`, and stored in the `api_keys` table.

Active keys include:

- `name` for operator-friendly identification.
- `role` for access control (`viewer`, `operator`, `admin`).
- `rate_limit_per_minute` for per-key throttling.
- `is_active` for revocation without deleting audit history.

Admin-only key management endpoints use `require_roles("admin")`.

## Key Rotation Procedure

1. Create a replacement key with an admin key: `POST /api/v1/api-keys`.
2. Move clients to the new plaintext key returned by the create response.
3. Confirm traffic with the new key.
4. Revoke the old key with `DELETE /api/v1/api-keys/{key_id}`.

Plaintext keys cannot be recovered from storage. Generate a new key if a client loses one.

## Rate Limiting

- Redis sliding-window limiter per API key.
- Window: 60 seconds.
- Limit source: `api_keys.rate_limit_per_minute`.
- On exceed: `429 Too Many Requests` with `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `Retry-After`.

## Demo Mode

When `DEMO_MODE=true`, the configured `DEMO_API_KEY` may access read-only demo flows. Non-GET requests are rejected with `403` so demo users cannot mutate records.

## Webhook Security

Webhook deliveries are signed with HMAC-SHA256:

- Header: `X-Signature-256`.
- Timestamp header: `X-Timestamp`.
- Retry schedule: immediate, 30 seconds, 5 minutes, 30 minutes.
- Permanent failures are pushed to the Redis dead-letter queue `dlq:webhooks`.

Webhook signing secrets can be encrypted at rest with AES-GCM using `AES_KEY` (base64-encoded 32-byte key).

## PII Trace Sanitization

`app/services/pii_sanitizer.py` redacts SSNs, credit cards, phone numbers, and email addresses before data is sent to external trace providers. The sanitizer recursively handles nested dictionaries and lists.

Guardrail checks also scan extraction outputs for PII and hallucination risk before review or downstream use.

## Data Handling

- Uploaded file bytes are stored through the configured storage backend.
- Extracted records and embeddings are stored in PostgreSQL and pgvector.
- LLM traces are used for cost tracking and debugging.
- Semantic cache entries use TTL-based expiry when enabled.
- Logs should include document hashes, job IDs, and request IDs rather than raw document text.

## Production Secret Requirements

Set these through the deployment secret manager, not committed files:

- `API_KEY_SECRET`: 32+ character secret for API key HMAC hashing.
- `ANTHROPIC_API_KEY`: model provider credential.
- `GEMINI_API_KEY`: embedding/judge provider credential where enabled.
- `DATABASE_URL`: PostgreSQL connection string; use SSL in production.
- `REDIS_URL`: Redis connection string; use TLS in production where available.
- `AES_KEY`: base64-encoded 32-byte key for encrypted webhook secrets.
- `LANGSMITH_API_KEY`, `LANGFUSE_PUBLIC_KEY`, `LANGFUSE_SECRET_KEY`: optional tracing credentials.

## CI Security

- `bandit` static analysis runs in CI.
- `ruff` and `mypy` gates catch common unsafe code paths and contract drift.
- Docker images are built in CI and published to GHCR on main.
- `python scripts/audit_portfolio_claims.py` keeps public security and portfolio claims aligned with committed evidence.
